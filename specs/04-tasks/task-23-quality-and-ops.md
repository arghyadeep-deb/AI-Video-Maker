# Task 23 — Post-Launch Quality Fixes & Ops Hardening

**Demo:** Reliable nightly backups, no silent video truncation, natural Hinglish narration with a real CTA — plus the groundwork for HuggingFace-model-based quality upgrades (image relevance, generated footage, voice expressiveness).

## Why this task exists

Task 22 finished the local HD engine build and deployed it live. Everything since has been real production operation on the live Oracle VM + Vercel frontend: bugs only visible under genuine live use, plus a new owner-driven initiative to fix three quality complaints (image relevance, voice monotony, Generated Footage reliability) using free HuggingFace infrastructure. This task collects that work since it doesn't fit any single task 01-22.

1. **Ops hardening** (done): nightly backup/retention services silently failing, worker-agent headless logging, SSH key rotation, official frontend link discovery, Abhijit's account reset.

4. **HuggingFace-model quality initiative** (in progress, see Completion notes): image relevance (CLIP or FLUX-generated images replacing stock-photo search), Generated Footage reliability (ZeroGPU-hosted image-to-video), voice expressiveness (Kokoro-82M as a possible edge-tts replacement). Scoped via brainstorming; two spike-tests (ZeroGPU video timing, Kokoro Hinglish handling) are the next step before committing to the bigger build.

## Completion notes

**Ops hardening (2026-07-16/17, live on Oracle VM):**
- `aivideomaker-backup.service` and `aivideomaker-retention.service` were both silently failing every scheduled run (`203/EXEC` — the deploy scripts had lost their executable bit, likely during a git checkout). Fixed permissions; also made `backup.sh`'s off-VM rclone copy step non-fatal (falls back to local-only backup honestly instead of failing the whole nightly job) since the R2 remote was never actually configured with real credentials — that's a genuine one-time owner action still open (needs the owner's own Cloudflare account).
- Worker-agent launched headless (`pythonw.exe` via Scheduled Task) had zero log output — stdout/stderr are discarded, not hidden, when there's no console to inherit. Added an explicit `logging.FileHandler` so future debugging isn't blind.
- The SSH private key used for VM access was displayed in a screenshot mid-session (Cloud Shell troubleshooting) — rotated immediately: new ed25519 keypair generated, old key's `authorized_keys` entry removed and verified rejected, new key verified working, old key file deleted locally.
- Discovered the **official frontend was never actually being used for real testing** — earlier "browser-verified" sessions were against a local dev frontend. The real, deployed, working frontend is `https://aivideomaker-app.vercel.app` (Vercel project `ai-video-maker`), confirmed via a real login + library load against the live backend.
- `abhijitdeebb@gmail.com`'s forgotten password: original was never recorded anywhere (DB only ever held the hash) — account deleted and recreated fresh with a new password, delivered to the owner to forward.
- Regular SSH to the Oracle VM remains intermittently unreachable for stretches of 10-30+ minutes (external network-path throttle, confirmed not a client bug — reproduced identically across two independent SSH clients). No fix found. An OCI Instance Console Connection was set up as a documented fallback path but proved unreliable in practice (Cloud Shell's FIPS-mode OpenSSH client can't use ed25519 keys at all — `identity file type -1`; a serial-console session also needs password auth the VM doesn't have configured). Regular SSH has always eventually recovered on its own within the session; no other fix attempted.

**Real bug: Generated Footage silently falling back to Ken Burns every time (2026-07-16/17):**
- Root cause #1: `worker_lease_timeout_s` was 35s against a 10s heartbeat interval — only ~2-3 heartbeat cycles of tolerance. The owner's home network (the same path SSH rides) has real multi-minute dropouts; a single missed heartbeat during scene_gen's long model-load window reliably blew the budget and the VM reclaimed the lease mid-generation. Fixed: raised to 120s (still fails over within ~2 min for a genuinely dead worker, but tolerates realistic jitter).
- Root cause #2 (deeper, not fixable by a timeout): even with the lease fix, `scene_gen` (Wan2.2-TI2V-5B + `enable_model_cpu_offload()`, required to fit the owner's 12GB laptop card) took **30+ minutes just to load the model**, confirming task-22's own bake-off finding ("impractically slow... after hours of cumulative attempts") independently, live. Added a `scene_gen`-specific 5-minute wait ceiling (`SCENE_GEN_WAIT_TIMEOUT_S` in `mode_b.py`, via a new `timeout_s` override param on `gpu_router.wait_for_task`) so a doomed attempt fails over to the working Ken Burns fallback in minutes, not 30+. Verified live, twice, with real timestamps: render completed end-to-end in under 6 minutes both times (previously: 30+ minutes or an indefinite hang).
- **The underlying reliability problem is not solved** — `scene_gen` on the home GPU worker is not currently practically viable. This is the direct motivation for the ZeroGPU-based approach below.

**Real bug: Mode B's final video silently cut off at the end (2026-07-17):**
- `stage_assemble`'s ffmpeg command used `-shortest`, which truncates the *entire* output to whichever mapped stream (video or audio) is even a single frame shorter — contradicting the stage's own documented design intent ("hold-last-frame tail, so audio always rules the timeline"). Verified by hand that the crossfade padding math (`clip_render_duration`) already nets out to exactly `sum(scene_durations)` by construction, so `-shortest` was pure downside: any harmless sub-frame rounding drift (zoompan's frame-count rounding, etc.) got silently converted into a truncated ending. Replaced with an explicit `-t {total_duration}` bound. Confirmed pre-existing test flakiness in `test_mode_b_footage.py` (a background-thread timing race in the `_fake_agent` test helper, reproduced identically with and without this change) is unrelated to this fix.

**Script quality: Hinglish + end-of-video CTA (2026-07-17, live-verified with real Gemini output):**
- `build_prompt()`'s rule #1 explicitly said "never romanize, never mix in translation" — directly causing formal/textbook Hindi output. Replaced with an explicit Hinglish instruction (English loanwords for modern/technical/abstract concepts, Hindi grammar/particles) for `language == "hi"`. First attempt was too weak (model still defaulted to formal words like "दिनचर्या", "समय सारणी"); strengthened with concrete before/after word examples ("रूटीन" not "दिनचर्या", "शेड्यूल" not "समय सारणी", etc.) and re-verified live — second attempt produced genuinely natural code-switching ("अपना रूटीन बनाएं", "शेड्यूल बनाएं और फोकस रखें").
- Added an explicit two-part closing-CTA rule (follow the account/channel + DM for a personalized detail, worded to fit the content's own persona rather than a fixed script) — verified live: "हमारे अकाउंट को फॉलो करें और... हमें डीएम करें" reads naturally and on-brand.
- The existing Devanagari validation gate (`_validate_scene_text`) needed no changes — it only checks for *presence* of Devanagari characters, not a full match, so Hinglish text already passed.

**HuggingFace-model quality initiative — status: in progress, not yet built:**
- Scoped via brainstorming (multi-turn design conversation, see PR/session history — no separate written spec doc yet, decision recorded here instead per this task's own completion notes convention).
