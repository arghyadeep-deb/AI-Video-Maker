# Task 11 — Talking-Head Engine

- **Depends on:** Tasks 05, 07, 10
- **Estimated effort:** 3 days (highest-risk task)

## Objective

`TalkingHeadEngine.render(portrait, wav) → mp4` with the hosted backend split: **Wav2Lip on the VM CPU (default, always available)** and **SadTalker on a Hugging Face ZeroGPU Space (quality option, daily GPU-slot budget)**. Plus an admin-only manual import path.

## Files

- `backend/app/engines/talking_head/{base.py,wav2lip_local.py,sadtalker_zerogpu.py}`
- `hf-space/` — Gradio ZeroGPU Space: SadTalker checkout, `render(portrait, wav) → mp4` endpoint; called via `gradio_client`
- `vendor/` — pinned-commit Wav2Lip, setup scripted (`scripts/setup_models.py` downloads weights, aarch64-compatible)
- `notebooks/render_avatar.ipynb` + `backend/app/api/jobs.py` `POST /jobs/{id}/import-render` — **admin** escape hatch for stuck jobs
- `backend/app/quota/gpu_budget.py` — GPU-seconds ledger (consumed by task-15's slot credits)

## Implementation

- Wav2Lip: static-image mode on CPU; honest UI note ("lips animated, head still"); benchmark on the 2-core ARM VM and record real minutes-per-video.
- SadTalker Space: `--still` flag, WAV 16 kHz, portrait 1024²; progress estimated from audio duration; Space call timeout + retry-once; on ZeroGPU quota error → fall back to Wav2Lip and refund the slot.
- Engine choice per render: default Wav2Lip; "HD avatar (uses 1 GPU slot)" option when budget remains.
- Pin exact commits + weight checksums in both the repo and the Space; setup scripts verify.
- **LongCat-Video Avatar 1.5 evaluation gate — owner priority**: realism is a stated product requirement ("must not feel fake"), so LongCat is the *preferred* HD engine, not a curiosity. Evaluate on the ZeroGPU Space against SadTalker on 3 test portraits (quality per GPU-second), and specifically test the **distilled/quantized Avatar 1.5 variants for fit on the 5070 Ti's 16 GB** — if it fits the home worker, frontier-realism avatars become the online-default. Record the verdict here; adoption also closes the R5 license risk (MIT vs SadTalker's research license). Tier expectation set in UI copy either way: Wav2Lip = "animated photo", SadTalker = "HD", LongCat = "realistic".

## Tests

- Unit: engine selection + slot-budget logic; ZeroGPU-failure → Wav2Lip fallback with refund; import-render validation (duration must match audio ±2 s, admin-only).
- Integration (10 s fixture audio + fixture portrait): Wav2Lip CPU render completes; ffprobe sanity. SadTalker Space test against a stub `gradio_client`; one live Space call in a manual profile.
- Live smoke: 30 s Hindi audio through both engines; lip-sync eyeballed side by side.

## Completion notes

- **The LongCat-Video Avatar 1.5 vs SadTalker judgment gate (specs/AGENT-PLAYBOOK.md gate #1) is genuinely blocked, not completed.** This needs real GPU inference of both models on real test portraits to compare "realism per GPU-second" and check whether quantized LongCat fits the 5070 Ti's 16GB — none of that hardware/deployment exists in this session (no ZeroGPU Space deployed, no access to the owner's GPU worker, which is itself a later task, 20a). Per the playbook's explicit instruction for the three judgment gates ("do NOT decide these alone... prepare the artifacts, present them, record the owner's verdict"), this is left unrecorded here for the owner to run once the SadTalker Space is deployed and LongCat is evaluated against it. **Owner action needed.**
- **Wav2Lip vendoring involved cloning a third-party research repo and running `torch.load` on downloaded pretrained weights** — a real code-execution/supply-chain risk category (unsafe deserialization, untrusted external code), which the auto-mode classifier correctly flagged and blocked on the first attempt. Paused and asked the owner directly rather than routing around it; proceeded only after explicit "allow it now" confirmation. Recorded here since it's a decision precedent: the *same* class of action for SadTalker's weights is deliberately left as an owner-run step in `hf-space/SETUP.md` rather than repeated in this session, since the SETUP steps run against Hugging Face's own build infrastructure, not this sandbox — a materially different risk profile (writing code that *will* run elsewhere later vs. actually running untrusted code *here*), but still flagging it explicitly rather than silently deciding it's fine.
- **Pinned Wav2Lip commit**: `bac9a81e63ecc153202353372e5724b83d9e6322` (Rudrabha/Wav2Lip), vendored at `backend/vendor/wav2lip/`. Research/non-commercial license — already an accepted risk (R5).
- **Weight provenance** (recorded in `backend/scripts/setup_models.py`'s own docstring too): `wav2lip_gan.pth` (435,801,865 bytes, sha256 `ca9ab7b7b812c0e80a6e70a5977c545a1e8a365a6c49d5e533023c034d7ac3d8`) from the Hugging Face mirror `camenduru/Wav2Lip` — the official Google Drive link throttles/blocks scripted downloads, a known problem with this exact repo. `s3fd.pth` (89,843,225 bytes, sha256 `619a31681264d3f7f7fc7a16a42cbbe8b23f31a256f75a366e5a1bcd59b33543`) from the URL Wav2Lip's own README links directly (the filename itself embeds the hash prefix, `s3fd-619a316812.pth` — the upstream project's own self-consistency convention, about as close to provenance verification as exists for a research checkpoint with no official published checksum).
- **Found and fixed a real, non-obvious bug in Wav2Lip's own `inference.py` by actually running it, not by reading the code**: its final ffmpeg-mux step built an unquoted shell command string (`'ffmpeg -i {} ...'.format(...)`) — breaks the instant any input path contains a space, and this project's own repo path (`AI video maker`) has one. Fixed by switching to a list-form `subprocess.call([...])` (no shell interpolation at all). This would have silently broken every single render on this dev machine's path layout.
- **2019-era dependency pins don't have Python 3.13 wheels** (`torch==1.1.0` etc.) — installed modern CPU-only `torch`, `torchvision`, `librosa`, `numba` instead, coexisting fine with `opencv-python-headless` (task-10). One real API-shape casualty: modern `librosa.filters.mel()` requires keyword arguments where the 2019 code passed them positionally — patched in the vendored `audio.py`.
- **`Wav2LipLocalEngine` shells out to `inference.py` via subprocess rather than importing it in-process** — Wav2Lip's modules carry global state (a cached mel basis, argparse-parsed globals) and the script hardcodes relative paths (`temp/result.avi`) that only resolve when cwd is the vendored repo itself. Subprocess isolation sidesteps both cleanly and matches the same "shell out to messy research code" pattern already used for ffmpeg elsewhere in this codebase.
- **Real, live-verified render**: a genuine ~9s WAV (synthesized via the already-working edge-tts engine from task-05) through the real, checkpoint-loaded Wav2Lip model against the same real test photo from task-10 (`tests/fixtures/test_face.jpg`) — 7.6s wall-clock on this dev machine's CPU, ffprobe-verified (both video+audio streams present, output duration within 1s of the source audio), and a visually-inspected extracted frame confirmed a real (not corrupted/garbage) rendered frame. **The production ARM VM's own timing is still unmeasured** — 2 cores, different architecture, no GPU; this dev-machine number is a sanity check that the pipeline *works*, not a queue-estimate input.
- **GPU-seconds ledger deliberately does NOT implement task-11's own older "per-user GPU slot credit" framing** — `specs/01-requirements/10-hosting-accounts-quotas.md` (locked, and explicitly written to supersede this: "quotas simplified... GPU budgeting inverts... Per-user credits are dropped") calls for a single global daily safety-rail counter instead, at this product's 1-2 user scale. Implemented as `app/quota/gpu_budget.py` reusing the existing `usage` table (date, counter, n) — the same pattern `api/script.py`'s `_increment_usage` already established for Gemini call counts — rather than adding a new per-user credits mechanism that a later, higher-authority requirements doc has already retired.
- **SadTalker engine's retry/refund logic distinguishes quota rejections from genuine crashes**: a ZeroGPU quota-exceeded error (pattern-matched from the exception text) refunds the pre-charged budget estimate and raises `ZeroGpuQuotaExhaustedError` so the caller (the new `chooser.render_with_fallback`) degrades honestly to Wav2Lip; any *other* failure (network blip, Space crash) keeps the charge (GPU time may genuinely have been spent) and retries once before giving up, per the task's own Implementation notes ("Space call timeout + retry-once").
- **Added `app/engines/talking_head/chooser.py`** (`render_with_fallback`) beyond the task's literal Files list — the task's own Tests section calls for "Unit: engine selection + slot-budget logic" as a testable unit, but no chooser module was explicitly named; this is the natural, minimal place for it, callable directly by task-12's `render_mode_a` pipeline once that exists.
- **Added an admin-role gate (`require_admin` in `core/deps.py`, a new `ForbiddenError` -> 403 handler) ahead of task-14's real auth** — the schema's `users.role` column already existed unused; the admin escape-hatch endpoint needed *some* gate now, so it checks that column directly rather than waiting for task-14. Test coverage flips the dev-mode seeded user's role explicitly to exercise both the authorized and rejected paths (there being only one fixed dev user pre-auth, this is the only way to test rejection without inventing a second real user).
